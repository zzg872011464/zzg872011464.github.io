第七章 Lambda表达式和流

发表于2024-06-16|更新于2024-06-16
|阅读量:

item 42. 与匿名类相比,优先选择Lambda表达式

除非参数类型的存在能让程序更清晰,否则应该省略Lambda表达式中所有参数的类型信息.如果编译器生成了一条错误,告诉我们它无法推导出某个Lambda参数的类型,那时再指明.

Lambda表达式没有名字,也缺乏文档;如果计算不是不言自明的,或者超过几行代码,就不要将其放在Lambda中.
使用Lambda表达式时,一行代码是最理想的,三行是合理的最大值

相较于Lambda表达式,匿名类的优势有 :

  1. 可以继承一些抽象类

  2. 可以访问当前实体的方法

item 43. 与Lambda表达式相比,优先选择方法引用

item 44. 首选标准的函数式接口

除非要增加的函数式接口满足以下三个特性中的一个或多个,不然应该使用默认提供的函数式接口

  1. 他将被频繁使用,一个描述性的名字可以带来好处(例如: Comparator)
  2. 它有严格的通用约定
  3. 自定义的默认方法能带来好处

如果编写函数式接口记得使用@FunctionalInterface注解

item 45. 谨慎使用流

流管道是延迟求值的:求值直到终结操作被调用时才会开始,并且永远不会计算对于完成终结操作而言并不需要的数据元素.

  1. 如果不调用终结操作那么中间操作不会触发

  2. 执行终结操作过程中,并不需要某些数据元素来完成这个操作,那么这些数据元素将不会被计算或处理.这种特性使得 Stream API 非常高效,因为它避免了不必要的计算,只处理真正需要的数据.

通常应该避免使用流来处理char值,java不支持CharStream

没有必要将所有的迭代(for,while等)全部改成流,虽然可以做到,但是很可能会损害代码的可读性和可维护性

流管道通过使用函数对象(通常是使用Lambda或者方法引用)来表达重复的计算,而迭代代码使用代码块表达重复的计算.通过代码块可以做到一些通过函数对象无法做到的事情

  1. 从代码块中,可以读取或修改作用域内的任何局部变量;而从Lambda中,只能读取final或者effective final的变量,并且不能修改任何局部变量
  2. 从代码块中,可以使用return,实现从包围的方法中返回,也可以在包围的循环内执行break或continue,或抛出异常;而从Lambda中,这些事情都不能完成

在以下场景,使用流可能会使工作变得非常容易

  1. 对元素的序列进行统一的转换
  2. 对元素的序列进行过滤
  3. 使用单个操作将这些元素组合起来(相加,连接,或计算最小值等)
  4. 将元素序列累加到一个集合中,或者是按照某种方式进行分组
  5. 在一个元素序列中查找满足某个条件的元素

一般来讲,终结操作中的forEach操作应该仅用于报告流计算的结果,而不是执行计算

三个参数的toMap方法,当存在冲突时,收集器会采用后写入者获胜策略,对于很多流而言,结果是不确定的

item 47. 作为返回类型时,首选Collection而不是Stream

流没有随机获取(类似数组索引)和循环多次这种概念

在读取文件等场景时,流不会像集合那样,一口气把全部的内容加载进来,而是会一条一套的加载,从而在文件特别大时避免产生OOM (在写工具类时可以分别提供流和集合两种返回方式)

item 48. 将流并行化时要谨慎

慎用parallel()方法,如果源流来自Stream.iterate或使用了中间操作limit,将这样的流管道并行化不太可能提高其性能

最适合并行化的终结操作是归约(如min,max,count和sum),短路操作(如anyMatch,allMatch和noneMatch)也适合并行化.
由Stream的collect方法执行的操作称为可变规约,他不适合并行化,因为对集合进行合并的开销很大.
